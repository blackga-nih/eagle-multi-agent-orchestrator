All errors are identical:
```json
{"level": "ERROR", "logger": "opentelemetry.exporter.otlp.proto.http.trace_exporter",
 "msg": "Failed to export span batch code: 401, reason: Unauthorized"}
```

**Root cause**: The backend QA deployment uses `LANGFUSE_PUBLIC_KEY` / `LANGFUSE_SECRET_KEY` to construct Basic auth for the OTLP endpoint (`https://us.cloud.langfuse.com/api/public/otel/v1/traces`). The startup probe at `server/app/strands_agentic_service.py:126-144` validates auth before registering the exporter. Errors appearing mid-flight means credentials became invalid after boot — likely a Langfuse API key rotation or expiration that happened during the April 13 window and has not been remediated.

**Impact**: Langfuse trace gaps for QA sessions. This is the **2nd consecutive day** of this issue (31 errors yesterday, 30 today).

| # | Timestamp (UTC) | Message |

| 1 | 2026-04-15 07:00:42 | `keepalive_ping: us.anthropic.claude-sonnet-4-6 FAILED: ServiceUnavailableException — Bedrock is unable to process your request` |
| 2 | 2026-04-15 07:02:25 | Same — circuit breaker notified, `OPEN (failures=2, threshold=2)` |
| 3 | 2026-04-15 07:14:23 | Same — circuit breaker `OPEN (failures=3, threshold=2)` |

**Root cause**: Bedrock returned `ServiceUnavailableException` for model `us.anthropic.claude-sonnet-4-6` during keepalive pings. The keepalive loop (`server/app/strands_agentic_service.py:625-677`, interval=45s) sends minimal `converse()` calls to warm the model. Three consecutive failures at 07:00, 07:02, and 07:14 UTC tripped the circuit breaker (threshold=2, config at line 467).

**Circuit breaker behavior**:
- After 2 failures: circuit opened at 07:02:25 (state: OPEN)
- After 3 failures: circuit re-confirmed OPEN at 07:14:23
- Recovery timeout: 300s (`EAGLE_CB_RECOVERY_TIMEOUT` env var)
- During OPEN state: requests cascade to fallback models (Sonnet 4.5 → Haiku)

**Impact**: Users between 07:00–07:19 UTC (est. recovery) would have been served by fallback models. The keepalive/circuit-breaker system worked as designed — this is a transient Bedrock availability issue, not an application bug.
