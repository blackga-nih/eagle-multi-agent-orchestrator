#### Detailed Error Evidence

**Error 1: OTel OTLP 401 Unauthorized (34 occurrences)**

All 34 errors originate from `opentelemetry.exporter.otlp.proto.http.trace_exporter` on log stream `5bf9ee03...` between 16:09–18:03 UTC on Apr 10. This is a slight decrease from 39 errors in the prior triage (Apr 9–10 window) but the pattern is identical. Every Langfuse span export fails with 401, creating a complete observability blind spot.

The startup probe in `server/app/strands_agentic_service.py:128-143` checks for 401 and skips registration. However, line 145–146 has a fallback path: if the probe throws any non-401 exception (timeout, DNS failure, connection error), the exporter is registered anyway (`logger.warning ... registering exporter anyway`). This is likely how the exporter got registered despite invalid credentials.

Root cause: The QA ECS task either (a) has different Langfuse keys injected via task definition environment that passed the initial probe but were later rotated/expired, or (b) the probe threw a non-401 exception allowing registration, and subsequent exports fail with 401.

**Errors 2–4: Bedrock ServiceUnavailableException + Circuit Breaker (5 events)**

Concentrated in a single 1-minute window at 07:00–07:01 UTC on Apr 10, all from the older ECS task `cc39c01b...`. Bedrock returned `ServiceUnavailableException` for `us.anthropic.claude-sonnet-4-6`. The circuit breaker correctly opened after 2 failures (threshold=2). This is a transient Bedrock-side issue — no code fix needed.

**Error 5: Missing Knowledge Base Template (1 occurrence)**

At 17:57 UTC, a user (ID: `64d8a488-...`) in session `4c1a026c-55b5-47e7-bafa-db17e9445a52` triggered a knowledge_fetch for `eagle-knowledge-base/approved/supervisor-core/essential-templates/Justification_and_Approval_Under_350K_Template.docx`. S3 returned `NoSuchKey`. This document is referenced by the agent templates but doesn't exist in the S3 knowledge base.

Code path: `server/app/tools/knowledge_tools.py:1403-1405`

**Error 6: Session Not Found (1 occurrence)**

At 17:46 UTC, same user (`64d8a488-...`) in session `7d98967a-d938-4f82-bcd9-f4a97bce7e67` triggered `set_active_package` but the session was not found in DynamoDB. This could indicate a race condition where `set_active_package` is called before the session is persisted, or the session expired/was deleted.

Code path: `server/app/package_context_service.py:127-130`
