| # | Timestamp (UTC) | Event |

| 1 | 2026-04-16 06:47:43 | `keepalive_ping FAILED: ServiceUnavailableException … reached max retries: 1` → `circuit_breaker → OPEN (failures=4, threshold=2)` |
| 2 | 2026-04-16 07:27:29 | Same error → `OPEN (failures=5, threshold=2)` |

HALF_OPEN probe transitions observed at 06:53:08, 07:34:01, 07:53:56, 08:12:05 — the recovery timer works. But the failure counter carries over across cycles.

**Root cause (AWS side)**: Bedrock returned 503 for Sonnet 4.6 during ~1.5h overnight. No app-side trigger; no user requests fell through to terminal failure (the fallback chain Sonnet 4.5 → Haiku was healthy).

**Root cause (app side)**: `ModelCircuitBreaker.record_failure()` at `server/app/strands_agentic_service.py:424-438` always increments `_failure_counts[model_id]` and never resets the counter on OPEN → HALF_OPEN transition at `server/app/strands_agentic_service.py:400-406`. Each subsequent failure just ratchets the count higher, producing misleading `failures=N` telemetry (N=4,5,6,7 here) and making it impossible to distinguish "first OPEN of a fresh incident" from "re-entry after transient failure."

**Impact**: Moderate — cosmetic/observability bug today, but breaks any alerting that keys on the failure-count delta per cycle, and obscures true post-recovery health. No user-visible error during the window.

| 6 | 2026-04-16 08:17:10 | `us.anthropic.claude-sonnet-4-6` | **305.0s** |

These latencies are clustered tightly around **300 s**, matching the `EAGLE_BEDROCK_READ_TIMEOUT` default at `server/app/strands_agentic_service.py:482`. The ping is returning via the `read_timeout`, not via a keepalive-specific deadline.

**Root cause**: `_ping_model()` at `server/app/strands_agentic_service.py:578-596` uses the shared `BedrockModel.client` with a 300-second `read_timeout`. When Bedrock is unresponsive, each keepalive ping hangs the loop for up to 5 minutes. The loop's own `_t.sleep(_KEEPALIVE_INTERVAL=45)` doesn't run during the hang, so the intended 45-second cadence degrades to one probe per ~5–6 minutes during an incident — exactly when we need *more* probing, not less.

**Impact**: Medium — during a Bedrock slow-down, the circuit breaker can't react promptly because its failure signal (keepalive) is itself slow. This amplifies the window where users are routed to a degraded primary before failover kicks in on the live request path.
